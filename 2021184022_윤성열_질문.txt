1. 현재 RIO로 서버를 만들었다. IOCP 같은 경우 Socket에 WSARecv, WSASend를 걸고 GQCS를 통해 입출력 완료 통지 알람을 받게 된다. RIO 같은 경우 RequestQueue에 RIoSend, RioRecv를 걸고, CompletionQueue에서 DequeueCompletion을 해서 입출력 완료 통지 알람을 받을 수 있다. 이때, RQ와 CQ는 Thread-Safe하지 않다. 따라서 세션을 특정 쓰레드에 고정해서 사용하는 걸로 알고있다. 이때, 쓰레드 당 관리해야 하는 세션의 개수는 기준점을 뭘로 둬야하는지.
  
 => 논문참조, https://scienceon.kisti.re.kr/srch/selectPORSrchArticle.do?cn=JAKO202007650437083 이것도 참조하면 좋다.

2. MO 게임도 최대 세션 개수를 정해놔야 하는지

  => 당연히 정해 놓는다 => MMO에서 최대 동접을 정해 놓고 그이상 받지 않는 것과 같은 이유.

3. MO게임 같은 경우 NPC의 업데이트 주기를 몇초로 하는게 적당한지, 현재는 Room의 Update 함수를 20ms마다 실행하고 있음, Room Update에서는 플레이어와 NPC들을 업데이트 하고 있음
  => 언리얼게임 같은 경우는 33ms 마다 업데이틑 한다. 15ms 마다 하기도 한다. 20ms 적당해 보인다.

4. MO게임의 경우, 이동동기화를 어떤식으로 해야하는지. 현재는 자기 클라이언트는 자기가 업데이트 하고, 남의 클라 정보만 서버에서 받아와서 보간하는 식으로 하고있는데 이 방식이 옳은 방식인지, 또한 이동 패킷을 서버로 줄 때, 어떠한 정보를 줘야 하는지

=> 그러한 방식으로 한다.  참조 (https://www.youtube.com/watch?v=h47zZrqjgLc)

5. 자기 클라이동도 서버에서 받아와야하는지, 아니면 자기 클라의 이동은 클라 안에서 자체적으로 처리하는지
 => 자기 이동은 클라에서 자체적으로 처리 후 서버에 이동 좌표를 보낸다. (일정 시간마다, 일정 거리 마다) => 서버에서는 이동에 문제가 없는 지 검증한다. (이동속도, 장애물여부, 플레이어 상태이상.....) => 문제가 있으면 클라이언트에게 옳바른 위치로 정정 packet을 보낸다. => 문제가 없으면 서버의 플레이어 객체의 좌표를 업데이트하고 브로드캐스팅한다.
